# SuperOPL Skill 开发日志（2026-05-13）

## 本阶段目标
验证 skill 在实际使用场景下的可用性，并根据发现的问题修正架构方向。

---

## 实际使用验证

### 场景
用户提供只读 API key，查询 task 8733932（静盘上料工具导致断刀），要求总结历史措施建议。

### 过程
1. 执行 `run.ps1 -Intent setup`：成功，OPL 322421（MOE1.1_L1_Morning Meeting OPL）验证通过。
2. 执行 `run.ps1 -Intent analyze -Query "静盘上料工具 断刀"`：失败，提示 index.json 不存在。
3. 执行 `run.ps1 -Intent knowledge-sync`：挂起，超过 10 分钟无响应，手动 Ctrl+C 终止。
4. 尝试让用户在本地终端手动跑 knowledge-sync：同样挂起，超过 10 分钟无响应。
5. 改用方案二：直接调用 API + PowerShell 脚本内存过滤，30 秒内成功拿到 task 8733932 完整数据及历史断刀记录。

---

## 发现的问题

### knowledge-sync 在当前环境不可行
- SuperOPL `/api/opls/{id}/tasks/` 无分页，必须一次性拉取全量数据
- OPL 322421 已有 2206 条记录
- knowledge-sync 脚本对每条记录做大量本地文件写入（wiki pages/chunks/entities/edges/evidence），整体处理时间过长
- 在公司网络环境（代理/VPN）下，持续超时或挂起
- 结论：**knowledge-sync 及依赖本地 index.json 的所有功能不可靠**

---

## 架构修正

放弃"先全量同步到本地知识库、再检索"的模式，改为**实时 API 查询 + 内存过滤**。

详细修订计划见：`plan/revised-plan-2026-05-13.md`

## 已完成开发

### WP-A：新增 `opl_query.ps1`
- 直接调用 API，在内存中按关键词/类型/负责人/owner/状态/时间范围过滤
- 支持 `-Keywords`、`-Type`、`-Responsible`、`-Owner`、`-Status`、`-DaysBack`、`-TaskId`、`-Top`
- 输出格式：table / json / markdown
- 通过 `run.ps1 -Intent query -Query "关键词"` 调用

### WP-B：改造 `opl_analyze.ps1`

- 改为实时拉取全量 API 数据，在内存中对每条任务按关键词评分排序

- 保留原有 table / json / markdown 输出格式
- 验证通过：查询"断刀"，2206 条数据中匹配 24 条，7 条 Problem，响应约 30 秒
